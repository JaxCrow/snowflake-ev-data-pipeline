# Implementation Plan: Rebuild Snowflake EV Pipeline

**Branch**: `001-rebuild-ev-pipeline` | **Date**: 2026-07-15 | **Spec**: specs/001-rebuild-ev-pipeline/spec.md

**Input**: Feature specification from `/specs/001-rebuild-ev-pipeline/spec.md`

## Summary

Rebuild the full EV data pipeline in a single Dev environment using a manual, stage-by-stage execution flow. The implementation includes mandatory NeonDB PostgreSQL CDC enrichment on a 12-hour cadence, end-to-end data quality verification, dbt validation, and analyst chat verification against expected curated metrics.

## Technical Context

**Language/Version**: Snowflake SQL + Snowpark Python 3.11 + dbt Core

**Primary Dependencies**: Snowflake (warehouse, tasks, streams, dynamic tables), Azure Blob Storage, NeonDB PostgreSQL, dbt-snowflake, Streamlit

**Storage**: Snowflake tables/schemas, external stage storage, Iceberg external volume

**Testing**: dbt tests, manual smoke validation, DQ validation procedures

**Target Platform**: Snowflake Dev account + Azure + NeonDB

**Project Type**: Data platform / analytics pipeline

**Performance Goals**:
- Bronze ingest trigger to load: target under 60s for test files
- Silver refresh cadence: near 1 minute target lag
- CDC enrichment cadence: 12-hour sync window
- Analyst validation queries: return expected metrics within normal interactive response times

**Constraints**:
- Dev-only deployment for first pass
- Manual setup required (no IaC in this pass)
- Mandatory external CDC enrichment
- Cost-optimized operation with resource monitoring and auto-suspend
- Completion gate requires SQL stages + dbt + DQ + Streamlit metric validation

**Scale/Scope**:
- Rebuild all defined pipeline stages in one pass
- Validate end-to-end lineage for representative EV records
- Cover full path from source ingestion to analyst-facing metrics

## Execution Guardrails (Constitution-Enforced)

- No stage execution proceeds without explicit user approval.
- Clarifications and approvals are handled one question at a time.
- Readiness gate verification is mandatory before any additional stage execution.
- Current execution boundary is readiness closure plus Stage 2 maximum unless explicitly re-approved.

## Go/No-Go Readiness Gate (Mandatory Before Stage Execution)

All items below must be verified and documented before any additional stage execution:

1. Azure resources and routing are operational (storage, queue, Event Grid path).
2. Snowflake authentication works for current run and role access is sufficient for Stage 1/2.
3. Integration trust and external volume access prerequisites are validated.
4. Secrets handling policy is compliant (no secrets in repository files).
5. CDC external connectivity prerequisites are available for later-stage execution.
6. Evidence logging destination is prepared for stage acceptance results.

Gate outcome rules:

- If any gate item fails, no new stage execution is allowed.
- If all gate items pass, execution is authorized only up to Stage 2 under current scope.
